# Безопасность публичного репозитория

Этот репозиторий **публичный**. Любой коммит в `main` может увидеть весь интернет.
Перед переносом проекта из `develop` и **перед сборкой EXE** обязательно пройдите чеклист.

## Чеклист перед переносом проекта

1. **Удалить секреты из файлов**
   - Нет файлов `.env`, `.env.local`, `.env.production` (только `.env.example` с плейсхолдерами).
   - Нет `credentials.json`, `service-account*.json`, `*.pem`, `*.key`, `*.p12`, `id_rsa`.
   - Нет паролей, токенов и API-ключей в коде, конфигах и комментариях.

2. **Проверить историю git**
   - Если секрет уже попал в коммит — одного удаления файла мало: ключ нужно **сменить/отозвать**.
   - При необходимости очистить историю (`git filter-repo` / BFG) и снова сменить ключи.

3. **Запустить автоматическую проверку**
   ```powershell
   cd C:\Users\420\Documents\develop\forgithub
   .\scripts\setup-security.ps1
   .\scripts\check-secrets.ps1
   ```

4. **Подготовить перенос из develop**
   ```powershell
   .\scripts\prepare-project-for-public.ps1 -SourcePath "C:\Users\420\Documents\develop\svyaz" -TargetName "svyaz"
   ```
   Скрипт не копирует `.env`, ключи и типичный мусор; после копирования снова запустите `check-secrets.ps1`.

5. **Коммит только после «зелёной» проверки**
   - Pre-commit hook блокирует коммит, если найдены секреты или запрещённые файлы.

## Чеклист перед сборкой EXE

1. Сборка только из папки внутри `forgithub` (не из `develop` с живым `.env`).
2. Переменные окружения для сборки — через CI или локальный `.env`, который **не** попадает в git.
3. В бинарник не должны попасть:
   - OAuth client secrets, JWT secrets, DB passwords;
   - приватные ключи TrueConf / облака / почты;
   - внутренние URL админок и IP, если они не предназначены для публики.
4. После сборки: `.\scripts\check-secrets.ps1 -Path dist` (или папка с EXE).
5. EXE и артефакты сборки (`dist/`, `build/`, `*.exe`) **не коммитить** в GitHub — только исходники.

## Что никогда не публикуем

| Тип | Примеры |
|-----|---------|
| Env-файлы | `.env`, `.env.local`, `.env.production` |
| Ключи | `*.pem`, `*.key`, `*.p12`, SSH-ключи |
| Облако | AWS keys (`AKIA…`), GCP JSON, Azure secrets |
| GitHub | `ghp_`, `gho_`, `github_pat_` |
| OpenAI / др. | `sk-…`, Bearer-токены в коде |
| Архивы с данными | `*.zip`, `*.rar` с бэкапами и дампами |

## Если секрет всё же утёк

1. Немедленно **отозвать/перевыпустить** ключ в сервисе.
2. Удалить секрет из репозитория.
3. При необходимости переписать историю git и force-push (осторожно, согласуйте с collaborators).
4. Проверить логи сервиса на подозрительную активность.
