# Mango Office + Jarvis: зачем нужен «публичный URL»

## В чём проблема

Jarvis у вас на компьютере слушает адрес **`127.0.0.1:8000`**. Это **локальный** адрес: его видит только ваш ПК.

Облачная АТС **Mango Office** работает в дата-центре в интернете. Когда звонит клиент, Mango должна **сама обратиться к вашему серверу**:

- «Запрос к внешней системе» в схеме звонка → HTTP на ваш URL  
- Уведомления о звонках → POST на `/api/telephony/webhook`

Если в Mango указать `http://127.0.0.1:8000/...`, Mango обращается **к себе в облаке**, а не к вашему ПК — **звонок не дойдёт до Джарвис**.

Это **не про «белый IP» на роутере** в классическом смысле. Даже с белым IP нужно:

1. Проброс порта 8000 на ПК (неудобно и небезопасно)  
2. Или **туннель** (рекомендуется): ngrok, Cloudflare Tunnel, localhost.run

## Схема

```
Клиент → Mango (облако) → HTTPS → ваш туннель → ngrok/cloudflare → 127.0.0.1:8000 (Jarvis)
```

Без среднего звена Mango **никогда** не увидит Jarvis.

## Решение (пошагово)

### Вариант A — ngrok (проще всего для теста)

1. Скачайте [ngrok](https://ngrok.com/), зарегистрируйтесь, добавьте authtoken.
2. Запустите Jarvis: `start-quick.bat` (окно «Jarvis — НЕ ЗАКРЫВАЙТЕ»).
3. В **другом** терминале:
   ```bat
   ngrok http 8000
   ```
4. Скопируйте **HTTPS** URL, например: `https://a1b2c3d4.ngrok-free.app`
5. В Jarvis → Панель разработчика → **АТС**:
   - Публичный URL: `https://a1b2c3d4.ngrok-free.app` (без `/` в конце)
   - Провайдер: **Mango Office**
   - Включено ✓ → Синтез речи → Сохранить
6. В личном кабинете Mango → схема входящего звонка → шаг **«Запрос к внешней системе»**:
   ```
   https://a1b2c3d4.ngrok-free.app/api/telephony/scenario
   ```
7. События (по желанию):
   ```
   https://a1b2c3d4.ngrok-free.app/api/telephony/webhook
   ```

Пока открыты **и** Jarvis, **и** ngrok — звонки работают. Закрыли ngrok — Mango снова «не видит» сервер.

### Вариант B — Cloudflare Tunnel (стабильнее для постоянной работы)

1. Установите `cloudflared`.
2. Настройте туннель на `http://127.0.0.1:8000`.
3. Получите постоянный поддомен `https://jarvis.ваш-домен.ru`.
4. Укажите этот URL в Jarvis и в Mango (как в шагах 5–7 выше).

### Вариант C — VPS (сервер в интернете)

Jarvis крутится на VPS с публичным IP и доменом — туннель не нужен.  
Для домашнего ПК обычно используют A или B.

## Настройка Mango Office (конкретно)

| Где в Mango | Что указать |
|-------------|-------------|
| Интеграция → API | API key + API salt → те же в Jarvis |
| Входящая схема → HTTP к внешней системе | `https://ВАШ-ТУННЕЛЬ/api/telephony/scenario` |
| Уведомления API (опционально) | `https://ВАШ-ТУННЕЛЬ/api/telephony/webhook` |
| IP-доступ к API (если включён фильтр) | Разрешить IP Mango: `81.88.80.132`, `81.88.80.133`, `81.88.82.36` |

Ответ Jarvis на scenario содержит URL аудио приветствия:
`/api/telephony/media/greeting.mp3`  
Полный адрес: `https://ВАШ-ТУННЕЛЬ/api/telephony/media/greeting.mp3` — его Mango проигрывает абоненту.

## Проверка без звонка

1. В Jarvis: **Тест webhook** — в логе должен быть `audio_url` с вашим туннелем.
2. В браузере откройте: `https://ВАШ-ТУННЕЛЬ/api/health` — должно быть `{"ok":true}`.
3. **Позвонить** (Mango API) — тест исходящего на ваш мобильный.

## Частые ошибки

| Симптом | Причина |
|---------|---------|
| Звонок без голоса Джарвис | Нет туннеля или неверный URL в схеме Mango |
| Таймаут в Mango | ngrok закрыт или Jarvis не запущен |
| `127.0.0.1` в настройках | Mango не достучится — только HTTPS туннель |
| Старый ngrok URL | После перезапуска ngrok URL меняется — обновите Mango и Jarvis |

## Итог

**Публичный URL** = адрес, по которому **Mango из интернета** достучится до Jarvis на вашем ПК.  
Практичное решение: **ngrok** или **Cloudflare Tunnel** + прописать URL в панели АТС и в схеме Mango.
